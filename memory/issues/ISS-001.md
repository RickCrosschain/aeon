# ISS-001 — Heartbeat recorded as failed on nearly every run

## Symptom

`memory/cron-state.json` shows heartbeat in a chronically broken state:

- `last_status: "failed"`
- `consecutive_failures: 54`
- `total_runs: 55`, `total_successes: 1`, `total_failures: 54`
- `success_rate: 0.02`
- `last_success: 2026-06-12T21:14:10Z` (the very first run; >36h stale → self-check fires)
- `last_failed: 2026-06-19T05:32:31Z`

Every heartbeat run since the first (2026-06-12) has been recorded as failed.

## Evidence

The recorded `last_error` is a truncated stream-json fragment:

```
ac6ff0f","total_cost_usd":0,"usage":{"input_tokens":0,"cache_creation_input_tokens":0,
"cache_read_input_tokens":0,"output_tokens":0,"server_tool_use":{"web_search_requests":0,
"web_fetch_requests":0},"
```

This signature is written **only** by the `Run skill` step in `.github/workflows/aeon.yml`
(~line 575–579), in the branch:

```bash
if [ "$CLAUDE_OK" != "1" ]; then
  echo "::error::All gateway providers failed ($CANDIDATES). Last output: $CLAUDE_OUTPUT"
  echo "$CLAUDE_OUTPUT" | tail -c 500 | tr '\n' ' ' | cut -c1-200 > /tmp/skill-error.txt
  exit 1
fi
```

So the Run step is treating the Claude call as a failed gateway attempt and `exit 1`-ing,
which the `if: always()` cron-state step (line 1030+) then records as `failed`.

## Analysis

The captured `CLAUDE_OUTPUT` tail is a **well-formed result object** (it contains the
`total_cost_usd`/`usage` envelope), but with **all token counts at 0**. That means the
invocation returned a structurally complete result while the gateway candidate's exit
code was non-zero — so `CLAUDE_OK` never gets set and the loop exhausts all candidates.

Notably the skill body still executes (this very heartbeat run produced real output and
regenerated `docs/status.md`), and the `if: always()` steps still commit. The breakage is
in **success detection / gateway exit-code handling**, not in the skill logic itself.

Two likely contributing factors, to be confirmed by skill-repair:
1. The active gateway provider returns content but a non-zero process exit code, OR
2. The provider returns a zero-usage result that the gateway wrapper (`scripts/llm-gateway.sh`)
   classifies as a failed attempt.

## Impact

- Fleet health metrics are wrong: the only enabled scheduled skill reads as critically failing.
- The self-check threshold fires on every run, masking genuine future failures.
- Token tracking, output capture, and quality analysis steps are all gated on
  `steps.run.outcome == 'success'`, so they are skipped on every run.

## Escalation — 2026-07-09

The condition worsened from intermittent to continuous. As of the 2026-07-05 08:12 UTC
heartbeat run, `consecutive_failures` was 0 and `last_status` was `success`. Since then:

- `consecutive_failures: 46` (was 0)
- `total_runs: 147`, `total_successes: 47`, `total_failures: 100`
- `success_rate: 0.32`
- `last_success: 2026-07-05T08:12:57Z` (~96h stale → **self-check >36h now fires**)
- `last_failed: 2026-07-09T08:40:23Z`
- `last_error`: same zero-token gateway signature (`total_cost_usd:0`, `input_tokens:0`,
  `output_tokens:0`).

Only **1 success in the last 47 runs**. No `memory/logs/` entries exist for 2026-07-06
through 2026-07-08, consistent with the success-gated output/log-capture steps being
skipped on every failed run. Root cause is unchanged (gateway exit-code / zero-token
success detection); the intermittent successes have simply stopped. Priority for
skill-repair raised accordingly.

## Suggested next steps (for skill-repair)

1. Inspect a recent heartbeat workflow run log to see the actual exit code and which
   gateway candidate(s) were tried (`vars`/`gateway.provider: auto`).
2. Check `scripts/llm-gateway.sh` for how it determines a candidate "succeeded" — confirm
   it isn't keying on a zero-token/zero-cost result or swallowing the real exit status.
3. Decide whether zero-usage-but-valid results should count as success, or whether the
   provider needs to be pinned/changed.
